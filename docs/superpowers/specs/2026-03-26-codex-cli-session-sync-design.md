# wecodex Codex CLI 会话同步设计

**目标**

让 `wecodex` 作为 Codex CLI 的远程入口，围绕同一套 Codex CLI 线程工作：微信端的 `/new`、`/list`、`/use N` 只是对 Codex CLI 线程的远程操作入口，不再维护一套独立会话。用户可以在微信和电脑之间查看、切换并继续同一项目目录下的历史线程。

---

## 1. 背景

当前 weCodex 已经能把微信消息转给本地 Codex CLI，但它更像“单次转发器”，而不是“会话入口”。这会带来两个问题：

1. 微信里和电脑上的 Codex CLI 不能自然共享同一段线程历史。
2. 用户无法在微信里查看和切换 Codex CLI 自己已有的历史线程。

本次需求的核心约束已经明确：

- 历史线程以 Codex CLI 为准
- `wecodex` 新开一个线程，本质上就是去 Codex CLI 新开一个对应线程
- `/list` 只列出**当前项目目录**下的 Codex CLI 线程
- `/use` 按 `/list` 展示的编号切换
- 微信和电脑同时操作同一线程时，按“自动合并上下文”处理，而不是互斥抢占

---

## 2. 用户体验目标

### 2.1 命令语义

微信端新增/强化的命令语义如下：

- `/new`
  - 在 Codex CLI 中新开一个线程
  - 新线程成为当前活跃线程
  - 微信和电脑端都指向这条新线程

- `/list`
  - 列出当前项目目录下 Codex CLI 可见的线程
  - 使用编号展示；编号只对**当前一次 `/list` 输出快照**有效
  - 当前活跃线程清晰标记出来

- `/use N`
  - 按 `/list` 里的编号切换当前活跃线程
  - 切换后，后续普通消息进入被选中的线程

- 普通消息
  - 发送到当前活跃线程
  - 微信里和电脑端看到的是同一条线程历史

### 2.2 范围约束

这里说的“当前项目目录”指的是**发起这次 `wecodex` 操作时，Codex CLI 当前绑定的项目工作目录**。如果当前 CLI 配置处于自动跟随模式，则它对应命令执行时的当前终端目录；如果是手动模式，则对应配置里持久化的工作目录。

`/list` 不显示全局线程，也不显示其他项目目录的线程。它只展示当前项目目录对应的 Codex CLI 线程集合，避免跨项目混淆。

### 2.3 冲突处理

当微信和电脑同时对同一线程发操作时，系统不应把它们拆成两条平行会话，而应尽量把消息都汇入同一个 Codex CLI 线程历史中。具体约束如下：

- 对同一线程的消息按接收顺序进入同一条线程历史
- `/new` 与 `/use N` 这类控制命令在写入当前活跃线程状态时是原子的
- 在控制命令生效之前已经进入队列的普通消息，仍按其被接收时对应的线程处理
- 不做双写、不做平行副本、不做抢占式断开

也就是说，这里处理的是“同一线程的多入口写入”，不是“多副本同步”。

---

## 3. 设计原则

### 3.1 以 Codex CLI 为唯一真相

weCodex 不维护自己的“主会话历史”。它只做两件事：

- 发现 Codex CLI 当前可见的线程
- 把微信侧命令和消息映射到这些线程上

这样可以最大限度避免 weCodex 和 Codex CLI 语义分叉。

### 3.2 只保留必要的本地状态

本地最多保留这些辅助信息：

- 当前项目目录下线程编号与线程 ID 的映射
- 当前活跃线程 ID
- 便于恢复的少量索引信息

本地缓存只用于提升可用性，不作为最终事实来源。

### 3.3 保持命令层薄

微信端命令解析层不直接理解 Codex CLI 的内部细节，只负责把 `/new`、`/list`、`/use N` 转交给线程适配层。

---

## 4. 核心架构

建议拆成三层：

### 4.1 线程仓库（thread registry）

职责：

- 读取当前项目目录下可见的 Codex CLI 线程
- 维护 `/list` 显示所需的编号映射
- 提供按编号查找线程、按线程 ID 查找线程的能力

它需要暴露的能力很小：

- 列出线程
- 根据编号定位线程
- 记录当前 active 线程

### 4.2 Codex CLI 适配器

职责：

- 调用 Codex CLI 的新建线程能力
- 读取线程历史
- 切换到指定线程
- 把普通消息发到当前线程

这层负责把“Codex CLI 能做什么”转换成 weCodex 能使用的统一接口。

### 4.3 命令层

职责：

- 解析 `/new`、`/list`、`/use N`
- 调用线程仓库和 CLI 适配器
- 将结果格式化成微信可读文本

命令层不保存线程真相，只负责路由和展示。

---

## 5. 数据流

### 5.1 `/new`

1. 微信端接收到 `/new`
2. 命令层调用 Codex CLI 适配器创建新线程
3. 适配器返回新线程 ID 和线程元数据
4. 线程仓库把它记为当前活跃线程
5. 微信端返回“已切换到新线程”之类的确认

### 5.2 `/list`

1. 微信端接收到 `/list`
2. 命令层向线程仓库请求当前项目目录下的线程列表
3. 仓库从 Codex CLI 可见线程中取出当前项目目录对应的集合
4. 返回按编号排序的列表文本

### 5.3 `/use N`

1. 微信端接收到 `/use N`
2. 命令层把 N 解析成列表编号
3. 线程仓库用编号找到目标线程
4. 适配器切到该线程
5. 仓库更新当前活跃线程
6. 后续消息发往该线程

### 5.4 普通消息

1. 微信端收到普通文本
2. 命令层把消息转交给当前活跃线程
3. 适配器把消息发到 Codex CLI 当前线程
4. 返回回复并保持线程继续生长

---

## 6. 线程编号策略

`/list` 使用稳定编号，但这个稳定性只在**同一次列表快照**内成立。也就是说：

- 编号用于人类操作 `/use N`
- 线程 ID 用于内部真实定位
- 当下一次 `/list` 重新生成快照后，编号可以重新分配，但线程 ID 不变

这是最简单、也最容易理解的交互方式。

---

## 7. 兼容与边界

### 7.1 只作用于当前项目目录

线程列表只展示当前项目目录下的线程。这里的项目目录以当前 `wecodex` 命令所绑定的 Codex CLI 工作目录为准。这样可以避免：

- 不同项目之间的线程串台
- 用户在微信里看到太多无关历史

### 7.2 不把 weCodex 做成独立聊天存储

本次不引入独立数据库来复制对话历史，也不做“双写一份微信聊天记录”。如果 Codex CLI 无法提供某些线程信息，weCodex 只在自身能力范围内做最小补充，不伪造完整历史。

### 7.3 自动合并而不是抢占

微信和电脑同时输入时，weCodex 的目标是把消息都送进同一条 Codex CLI 线程中，而不是让某一端失去控制权。实现上按单线程 FIFO 处理同一线程的消息；控制命令更新活跃线程状态时是原子的，但不会打断已经进入处理队列的普通消息。

---

## 8. 错误处理

建议的错误策略是：

- Codex CLI 不可用时，明确提示 CLI 不可用
- 当前项目目录下没有线程时，`/list` 给出空列表提示
- `/use N` 超出范围时，提示编号无效
- 新线程创建失败时，直接返回失败原因，不伪造切换成功
- 读取线程列表失败时，保留可观察的错误摘要，便于排查

错误信息应尽量短、明确、适合微信展示。

---

## 9. 测试策略

应优先覆盖这些行为：

### 9.1 命令语义

- `/new` 会调用 Codex CLI 新建线程
- `/list` 只返回当前项目目录线程
- `/use N` 按编号切换线程
- 普通消息发送到当前活跃线程

### 9.2 范围与映射

- `/list` 不混入其他项目目录线程
- 编号变化不影响线程 ID 的内部定位
- 切换后当前活跃线程更新正确

### 9.3 冲突与共享

- 微信里和电脑里对同一线程的操作都落到同一线程记录里
- 不会因为多入口而分裂出两条平行会话

### 9.4 错误路径

- CLI 不可用时返回可读错误
- `/use` 越界时返回可读错误
- 空列表时给出明确提示

---

## 10. 成功标准

满足以下条件即可认为本次设计目标达成：

1. 微信里可以通过 `/new` 直接创建 Codex CLI 线程
2. 微信里可以通过 `/list` 查看当前项目目录下的 Codex CLI 历史线程
3. 微信里可以通过 `/use N` 切换到指定线程
4. 普通消息会继续发送到当前活跃线程
5. 微信里和电脑端可以围绕同一条 Codex CLI 线程继续对话
6. weCodex 不再维护一套独立的“假会话”历史

---

## 11. 不在本次范围

以下内容暂不纳入本次工作：

- 跨项目聚合所有线程
- 云端同步线程数据库
- 多设备冲突仲裁策略面板
- 线程重命名、删除、归档
- 自定义过滤条件
- 自动探测并合并不同用户/不同账号的会话

这些都可以在后续作为独立能力继续扩展。
